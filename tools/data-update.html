更新规则

  1. 新增赛季或新天赋版本时，创建新的 data/seasons/* 目录。
  2. 新增版本先用 create-version.mjs 克隆草稿，确认后才写入。
  3. 脚本默认不写入文件，必须显式使用 --write。
  4. 已有源文件默认不覆盖；确实要替换时再加 --force。
  5. 新旧天赋 raw 数据用 diff-tree.mjs 比对后，再决定哪些路线要复核。
  6. 切换默认版本前，用 generate-version-log.mjs 记录 diff 和检查结果。
  7. 切换默认版本必须先跑 switch-current.mjs dry-run。
  8. 手调 BD 路线和市场快照不由基础源数据更新自动覆盖。

BD 数据要求

  • 标题、职业和升华保留简体、繁体、英文。
  • 阶段必须能对应天赋路线，节点数和覆盖策略要和 manifest 一致。
  • 攻略正文必须有 guideStrategy：核心循环、变强信号、卡关诊断和取舍规则。
  • 技能本体和辅助宝石分开；推荐链接只能引用已登记技能和辅助。
  • 装备优先级、阶段装备、制作计划要覆盖开荒到毕业。
  • 新增 BD 先用 scaffold-build.mjs 从模板生成草稿；脚本不会自动登记 manifest。
  • 未登记路线先用 scaffold-route.mjs 生成候选 JSON；节点 ID 必须来自人工确认的阶段路线。
  • 草稿补完整后再用 register-build.mjs 预览 manifest entry；确认后才 --write。
  • validate-all.mjs 会 dry-run 检查 BD 脚手架、路线脚手架和注册脚本，确认不会误写草稿、路线或 manifest。
  • 候选 BD 先用 --id 和 --file 单独校验，再登记到 manifest。

真实快照要求

  • source.type 使用 real-snapshot-v1。
  • 记录 source.provider、source.league、source.fetchedAt。
  • 记录异常价格过滤规则和最低挂单深度。
  • 每个条目带 marketEvidence，包含买卖挂单数量、观察时间、可信度和深度备注。
  • 原始采集结果先用 normalize-market-snapshot.mjs 生成候选快照。
  • normalize-market-snapshot.mjs 默认 dry-run，只预览候选结构和校验结果，不写入候选队列。
  • --write 写入候选后会更新 market/candidates/index.json，数据更新页会读取候选预检结果。
  • 候选快照先用 --file 和 --kind 单独校验，再登记到 manifest。
  • 正式归档使用 archive-market-snapshot.mjs dry-run，确认后再 --write。
  • 接入后运行 node scripts/validate-market.mjs --season s05。
